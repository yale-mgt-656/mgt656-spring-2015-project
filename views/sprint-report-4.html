Sprint 4 Retrospective

What went well during the sprint cycle?
  1. We fixed several pieces of code on the Home and About pages so the tests pass. This was because we were more focused on coding up the features rather than checking every single test.
  2. We used branches and pull requests and thus had no issues with merging code, unlike the previous sprint.
  3. We managed to finish some tasks in less time than expected such as (deleting old events) and we used this time to start the unexpected new tasks (Google Analytics tracking code).
What went wrong during the sprint cycle?
  1. We spent more time on making tests pass than on coding new things
  2. We are having trouble and feeling a bit desperate because some functions are working but still the test doesn’t pass and we don’t know why (form validation). We have had to give more time to this task in the next sprint which was unplanned.
  3. The new tasks added in the middle of our workload created a disruption to the plans we already had. As you can see in the graph the expectation is that we will not be able to finish the project on time (Pessimistic estimation)
What could we do differently to improve?
  1. From the beginning of our coding we should make sure that tests pass so we don’t lose extra time fixing code later
  2. Create more pull requests even if they are small commits instead of committing a whole chunk of code at the end
  3. Agree on meeting at least once a week
  4. We should be better prepared to receive more unexpected tasks from the stakeholders during our process

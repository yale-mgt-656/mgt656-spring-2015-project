Executive summary

RSVPd is an event platform where users can create their own events, RSVP using their Yale email addresses and browse through events which will be happening in the future.

In sprint 5, we finished all the required user stories for our project and ensured the following - i) all mocha tests are passing, ii) we have implemented the extra user stories on A/B testing, and iii) we added extra user stories to make our project and web page better (in terms of aesthetics and database connectivity) in the final sprint.

Product backlog

In Sprint 5, we completed the following activities:

  1. Created a donation button on the event detail page which links to a donation page
  2. Created a thank you page after a person enters an amount in the donation page
  3. The new event form is validated
  4. If the form is validated it is redirected with a 302 response
  5. Added RSVP text in the event box
  6. Made A/B testing code using Google Analytics
  7. Tested and analyzed the RSVP text
  8. Tested the donation button v/s support button
Sprint Retrospective

What went well during the sprint cycle?:

  1. We finished implementing all the user stories for the project!
  2. We ensured all the mocha tests pass.
  3. We worked on our own branches for each story and created pull requests which made it much easier to manage the project without merge conflicts
  4. We were able to implement Google Analytics A/B tests using documentation on its website, though it took up a lot of time and was not very straightforward to implement and test
  5. We discovered a bug in the donation class name (in Professor Kyle's code) due to which we weren't seeing conversions on our donate/support buttons.
What went wrong during the sprint cycle?
  1. The Google analytics were a challenge because we didn’t know how to use them and the documentation was not very straightforward
  2. The RSVP button added in each event box in the homepage could not be a button with a link. Whenever we added a link, it would fail the mocha tests. So we had to change the RSVP to plain text (refer to Experiment 1 images below)
  3. The Thanksgiving holiday prevented us from meeting as usual during the week, but we managed to work virtually from different locations
  4. We were very close to finishing all the tasks down to 6 story points, but we decided to add more tasks (as seen in the spike on the burndown chart) such as the style for the donation and the event detail pages etc.
What could we do differently to improve?
  1. We had created a user story for the RSVP button and implemented it without checking if all the tests run. We should have been more careful in understanding test case requirements by going through the test.js code.

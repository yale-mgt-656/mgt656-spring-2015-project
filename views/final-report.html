Executive Summary

RSVPd is an event platform that allows users to post events of different kinds - be it to relax, socialize, visit or party! Users can create their own events, RSVP using their Yale email addresses and browse through other events that have been posted. We built the site "standing on the shoulders of giants" and it uses Nunjucks, a Javascript templating system, and Node.js, a server side Javascript environment. We implemented Google Analytics code using which we conducted several experiments to determine which variation of our website was more appealing to users. We also had a testing framework using mocha to ensure the completion of user stories as per project requirements. The following points are some of the key characteristics that we believe make Eventbrite a [horrible] platform compared to us:

  1. An integrated and highly motivated team is behind the RSVPd website. Women comprised of 75% of the team, thus proving that women can code pretty darn well! Cheers to diversity.
  2. We created mockups using Balsamiq using Z patterns to make the user interface intuitive.
  3. We tested our website often and made sure all our tests pass!
  4. We have been able to view traffic patterns on Google Analytics and have received more than 8k sessions from Yale Law, Yale School of Management, and Yale Search!
  5. We implemented a RSVP/no-RSVP option and Donate/Support button and tested out what users find more compelling in order to deliver the best user experience.
  6. The website is simple and easy to use with all the required functionality. Enough said.

Variation 1

The first variation that we tested was if the word “RSVP” placed on the event box would increase the click-through rate in comparison with not having this word. Our original version is without the “RSVP” word and our Variation 1 is with the “RSVP” word as shown in the following pictures


Original: Without RSVP

Version 1: With RSVP

For this test we can say that our having the “RSVP” word increases the click-through rate by 1.94% and has a 50% probability of outperforming the original.

The Agile way

We structured our sprints so as to follow the practices discussed in class and ensured that we did things the Agile way.

Agile components What we did
Sprint planning At the beginning of each sprint, we scheduled time on our calendars for an in-person meeting. Here we looked at our product backlog on Waffle IO and each of us picked up the tasks we wanted to do keeping in mind the number of hours estimated for the task. We also discussed dependencies between tasks and collaborative efforts to complete them.
Scrum meeting We had 10 minute stand-up meetings during class time to discuss what we had finished, what we planned to do and if we were facing obstacles. If the obstacles were complicated, we used class time to go and talk to Kyle or the TAs.
Sprint demo We had a sprint demo on the Monday before every sprint report submission. This would ensure that we have working and usable code that pass the mocha tests corresponding to the user stories that we had completed.

Overall Retrospective

During the overall project, the team grew together by overcoming different challenges at various stages of the project. Besides, we were able to understand and respect everyone’s role into the project and create a webpage which we are very proud of. The thing that worked the best was planning and estimating the tasks, dividing the labor and then putting it together using the version control tools we learned during the course. Our top learnings from this course were:

  • We learned how to estimate tasks so that we can comfortably finish a project in 6 sprints rather than do a whole lot of work at the last minute (which some people - not us - would do in a typical b-school project).
  • We were able to pick up tasks that we liked to do most of the time. We also picked up routine tasks (such as creating HTML sprint reports) on a rotational basis which was a fair division of work.
  • Due to the visual and challenging nature of the project, we kept ourselves self-motivated.
  • The Waffle IO interface helped us to keep track of tasks very easily and was a very handy tool during sprint planning sessions.
  • We constantly helped each other during the entire projects - be it in person, over the phone or through screen-sharing sessions.
  • All of us developed our skills over the course of 6 weeks - through self-learning and through our team members!
  • The Google Analytics bit threw us off balance but we responded to change in a very positive way through planning and taking on extra work when needed.

During the process of learning we dealt with some difficulties such as spending too much time looking for a coding solution on our own, which eventually led to tasks slipping into the next sprint. However, every time we faced difficulties we discussed it as a team and if we weren’t able to find a solution, we took help from the TAs through the super helpful review sessions.

In general, we believe that this project helped us not only to develop team skills but to learn methodologies to develop efficient projects. To sum up, it taught us Agile values - interacting with the team, dealing with change, collaboration and making sure our code works!
